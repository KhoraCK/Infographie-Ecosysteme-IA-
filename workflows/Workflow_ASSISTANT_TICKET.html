1. Vue d'ensemble

1.1. Mission de l'Assistant Ticket

L'ASSISTANT TICKET est un workflow autonome qui surveille les nouveaux tickets GLPI et traite automatiquement les demandes de création de compte (première procédure implémentée). Il libère les techniciens des tâches répétitives tout en garantissant un contrôle humain via Safeguard.

Objectif

"Qualifier, traiter et exécuter les demandes de création de compte automatiquement, en vérifiant les autorisations et en utilisant le script existant."

Périmètre actuel (Phase 1)

Procédure Statut Description
Création de compte AD Implémentée Utilise le script d'Alexis via RDM (L3)
Suppression de compte AD Implémentée Exécution différée avec délai (L4)
Reset mot de passe Non implémenté Nécessite appel téléphonique (hors scope)
Déblocage compte Non implémenté Nécessite appel téléphonique (hors scope)
Limitation actuelle : Les resets de mot de passe et déblocages de compte ne peuvent PAS être automatisés car ils nécessitent un appel téléphonique pour transmettre le nouveau mot de passe à l'utilisateur.

2. Orchestrateur (Python)

2.1. Cycle de polling GLPI (180 secondes)

L'orchestrateur s'exécute en boucle infinie avec un intervalle de 180 secondes (3 minutes).

Cycle de l'orchestrateur

while True:
    # 1. Récupérer les nouveaux tickets GLPI
    tickets = glpi_api.search_tickets(
        status=["Nouveau", "En attente"],
        categories=["creation_compte", "ouverture_session"],
        age_max_hours=24
    )

    # 2. Pour chaque ticket
    for ticket in tickets:

        # 3. Vérifier si déjà traité (cache Redis)
        if redis.exists(f"assistant:processed:{ticket.id}"):
            continue  # Déjà traité, passer au suivant

        # 4. Détection mots-clés dans titre/description
        keywords = ["créer un compte", "ouverture de session",
                    "nouveau compte", "création utilisateur"]
        if not any(kw in ticket.content.lower() for kw in keywords):
            continue  # Pas une demande de création de compte

        # 5. Construire le contexte pour l'agent
        context = {
            "ticket_id": ticket.id,
            "ticket_title": ticket.title,
            "ticket_content": ticket.content,
            "demandeur_id": ticket.requester_id,
            "demandeur_name": ticket.requester_name,
            "entite_id": ticket.entity_id,
            "entite_name": ticket.entity_name,
            "categorie": ticket.category,
            "date_creation": ticket.date_creation
        }

        # 6. DÉCLENCHER L'AGENT
        result = agent_assistant.process(context)

        # 7. Marquer comme traité dans Redis (TTL 7 jours)
        redis.setex(f"assistant:processed:{ticket.id}", 604800, "1")

    # Attendre 180 secondes avant le prochain cycle
    sleep(180)

2.2. Filtrage et qualification préliminaire

L'orchestrateur effectue 4 vérifications déterministes avant de déclencher l'agent.

Vérification 1 : Cache Redis

Clé assistant:processed:{ticket_id}
TTL 7 jours
Logique Si la clé existe → ticket déjà traité → ignorer

Vérification 2 : Statut du ticket

Statuts acceptés Nouveau, En attente
Logique Tickets déjà assignés ou en cours → ignorer

Vérification 3 : Catégorie

Catégories acceptées creation_compte, ouverture_session, gestion_utilisateurs
Logique Seules ces catégories déclenchent l'agent

Vérification 4 : Mots-clés

Mots-clés recherchés "créer un compte", "ouverture de session", "nouveau compte", "création utilisateur"
Logique Au moins un mot-clé doit être présent dans titre ou description

2.3. Déclenchement de l'agent

Si toutes les vérifications passent, l'orchestrateur construit un contexte JSON et appelle l'agent.

3.2. Vérification du profil demandeur

C'est la première vérification critique de l'agent. Un utilisateur "lambda" ne peut PAS demander une création de compte.

Règle métier absolue : Seuls les profils Directeur et Référent peuvent demander une création ou suppression de compte AD.

Raisonnement de l'agent

// Contexte reçu
demandeur_id = 142
demandeur_name = "Marie MARTIN"

// Étape 1 - Récupérer le profil
Agent: glpi_get_user_profile(user_id=142)
Résultat: {
    "user_id": 142,
    "name": "Marie MARTIN",
    "profile": "Directeur",   // ← PROFIL CLÉ
    "entity": "MFI - Maison Familiale"
}

// Étape 2 - Vérification
Agent: "Le profil est 'Directeur'. C'est un profil autorisé.
        Je peux continuer le traitement."

PROFILS_AUTORISES = ["Directeur", "Référent"]
if profile in PROFILS_AUTORISES: CONTINUE
else: REJECT

Cas de rejet

Si le demandeur n'est pas autorisé

Résultat: {
    "profile": "Utilisateur",    // ← NON AUTORISÉ
}

Agent: "Le profil 'Utilisateur' n'est pas autorisé à demander
        une création de compte. Je dois rejeter la demande."

// Action: Ajouter un suivi explicatif
glpi_add_followup(ticket_id=5234, content="""
Bonjour,

Votre demande de création de compte ne peut pas être traitée.

Seuls les Directeurs ou Référents sont habilités à demander
la création d'un compte utilisateur.

Veuillez contacter votre responsable pour qu'il fasse la demande.

Cordialement,
Assistant WIDIP
""")

3.4. Identification du serveur AD

L'agent détermine le serveur Active Directory cible à partir du nom de l'entité.

Convention de nommage des serveurs AD

{NOM_CLIENT}-DC01

Exemples:
  MFI     → MFI-DC01
  ODYNEO  → ODYNEO-DC01
  ADAPEI  → ADAPEI-DC01

Raisonnement de l'agent

// Contexte
entite_name = "MFI - Maison Familiale Isère"

// Extraction du code client
Agent: "L'entité est 'MFI - Maison Familiale Isère'.
        Le code client est le premier mot avant le tiret: 'MFI'"

client_code = "MFI"

// Construction du nom du serveur AD
serveur_ad = f"{client_code}-DC01"  // = "MFI-DC01"

// Vérification de l'existence (optionnel)
Agent: ad_check_server(server="MFI-DC01")
Résultat: {exists: true, ip: "192.168.10.1"}

Agent: "Le serveur MFI-DC01 existe. Je vais exécuter le script dessus."
Rappel architecture AD : L'Active Directory est au niveau de l'ASSOCIATION (client), pas de l'établissement. Tous les établissements d'une même association partagent le même AD.

Exemple : Odyneo (15 établissements) → 1 seul AD : ODYNEO-DC01

4.3. Le script d'Alexis

La création de compte utilise un script PowerShell existant, développé par Alexis, qui automatise toutes les étapes de création.

Caractéristiques du script

Propriété Valeur
Langage PowerShell
Emplacement Sur chaque serveur AD ({CLIENT}-DC01)
Accès Via RDM (Remote Desktop Manager)
Paramètres Nom, Prénom, (optionnel: Poste, Groupes)

Appel du script par l'agent

// Paramètres préparés par l'agent
params = {
    "server": "MFI-DC01",
    "nom": "DUPONT",
    "prenom": "Jean",
    "login": "j.dupont",
    "email": "j.dupont@mfi.fr",
    "poste": "Éducateur spécialisé"
}

// Appel MCP (niveau L3 - validation requise)
Agent: ad_create_user(params)

// Le MCP encapsule l'appel au script
// Connexion RDM → MFI-DC01 → Exécution script Alexis

Résultat: {
    "status": "success",
    "login_created": "j.dupont",
    "password_temp": "MFI2026!Temp",  // Généré par le script
    "must_change_password": true
}
Sécurité : Le mot de passe temporaire généré est transmis au demandeur (Directeur/Référent) qui le communiquera au nouvel utilisateur. L'utilisateur devra le changer à la première connexion.

4.5. Procédure : Suppression de compte AD (L4)

La suppression de compte est une action critique et irréversible. Elle bénéficie donc du niveau L4 (CRITICAL) avec une exécution différée.

Niveau L4 - Exécution différée : Contrairement au L3 (validation immédiate), le L4 permet au technicien de choisir un délai avant l'exécution effective (1h à 48h). Cela offre une fenêtre de rétractation en cas d'erreur.

Différence L3 vs L4

Niveau Création compte (L3) Suppression compte (L4)
Validation Immédiate par technicien Immédiate par technicien
Exécution Immédiate après validation Différée (1h à 48h)
Annulation Impossible (sauf suppression) Possible pendant le délai
Réversibilité Élevée (on peut supprimer) Nulle (données perdues)

Flux de suppression avec délai

AGENT
=====
[1] Vérification profil demandeur (Directeur/Référent)
[2] Identification du compte à supprimer
[3] Appel ad_delete_user(...) → L4

SAFEGUARD L4
============
[4] Notification technicien dans WIBOT
[5] Technicien voit la demande + détails du compte
[6] Technicien choisit le délai:
    - 1h  (minimum - suppression rapide souhaitée)
    - 4h  (délai par défaut)
    - 24h (fin de journée)
    - 48h (délai maximum)
[7] Confirmation avec délai choisi

PENDANT LE DÉLAI
=================
[8] Action planifiée visible dans WIBOT
[9] Possibilité d'ANNULER à tout moment
[10] Rappel automatique 30 min avant exécution

EXÉCUTION
==========
[11] Si non annulée → Suppression effective du compte
[12] Notification de suppression → demandeur + technicien
[13] Clôture du ticket GLPI
Avantage du L4 : Si le demandeur réalise une erreur ("Non, pas ce compte !"), le technicien peut annuler l'action pendant le délai. Une fois le délai écoulé, la suppression est définitive.

## CONTEXTE
Tu reçois un contexte JSON de l'orchestrateur contenant:
- ticket_id: ID du ticket GLPI
- ticket_title, ticket_content: Contenu du ticket
- demandeur_id, demandeur_name: Demandeur
- entite_id, entite_name: Entité/Association

## TA MISSION
1. Vérifier le profil du demandeur (CRITIQUE)
   SEULS LES PROFILS Directeur, Référent SONT AUTORISÉS
   Si profil non autorisé → REJETER avec message explicatif

2. Extraire les informations du nouveau compte
   - Nom (obligatoire)
   - Prénom (obligatoire)
   - Poste (optionnel)
   Si nom/prénom manquants → Demander complément via suivi

3. Identifier le serveur AD cible
   - Extraire le code client de entite_name (premier mot avant "-")
   - Construire: {CODE_CLIENT}-DC01
   - Exemple: "MFI - Maison Familiale" → MFI-DC01

4. Exécuter la création (L3 - validation requise)
   - Appeler ad_create_user avec les paramètres
   - Attendre validation Safeguard

5. Documenter et notifier
   - Ajouter suivi GLPI avec login créé
   - Envoyer email au demandeur (login + MDP temporaire)
   - Clôturer le ticket

## OUTILS MCP DISPONIBLES
GLPI (L0-L2):
  - glpi_get_user_profile(user_id) → Profil du demandeur [L0]
  - glpi_add_followup(ticket_id, content) → Ajouter suivi [L1]
  - glpi_close_ticket(ticket_id) → Clôturer [L2]

Active Directory (L0-L4):
  - ad_check_server(server) → Vérifier serveur AD [L0]
  - ad_create_user(server, nom, prenom, login, email) → Créer compte [L3]
  - ad_delete_user(server, login) → Supprimer compte [L4 - différé]

Notifications (L2):
  - send_email(to, subject, body) → Email [L2]

## PROFILS AUTORISÉS
AUTORISÉS: ["Directeur", "Référent"]
NON AUTORISÉS: ["Utilisateur", "Technicien", autres]

## RÈGLES CRITIQUES
- TOUJOURS vérifier le profil AVANT toute action
- TOUJOURS extraire nom ET prénom avant création
- Le serveur AD = {CODE_CLIENT}-DC01 (depuis entite_name)
- Si profil non autorisé → Suivi explicatif + NE PAS TRAITER

9.3. Mécanisme d'ajout de procédure

Quand l'Enrichisseur crée une nouvelle procédure

1. L'Enrichisseur détecte un pattern récurrent dans les tickets résolus

2. Il génère une procédure structurée :
   {
     "id": "proc-ad-group-001",
     "titre": "Ajout utilisateur à un groupe AD",
     "mots_cles": ["ajouter groupe", "accès dossier", "permissions"],
     "profils_autorises": ["Directeur", "Référent"],
     "etapes": [
       "Vérifier profil demandeur",
       "Identifier utilisateur et groupe cible",
       "Exécuter ad_add_user_to_group()",
       "Documenter dans GLPI"
     ],
     "outils_mcp": ["ad_add_user_to_group"],
     "niveau_safeguard": "L2"
   }

3. La procédure est indexée dans le RAG

4. L'Assistant Ticket peut désormais traiter ce type de demande
