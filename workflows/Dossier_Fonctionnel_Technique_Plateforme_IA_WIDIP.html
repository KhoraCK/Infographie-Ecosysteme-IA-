Préambule

Ce document constitue le dossier fonctionnel et technique de la plateforme IA Support conçue pour WIDIP.

Cette plateforme vise à transformer le support informatique réactif en support proactif, en s'appuyant sur l'intelligence artificielle pour :

  • Assister les techniciens dans leurs tâches quotidiennes
  • Automatiser la surveillance des infrastructures clientes
  • Capitaliser sur les connaissances et l'expertise de l'entreprise
Note : WIDIP est l'entreprise pour laquelle cette architecture a été conçue. La plateforme elle-même (WIBOT + Workflows) n'a pas de nom spécifique à ce jour.

1. Objectif et vue globale

1.1. Objectifs de la plateforme

La plateforme IA Support répond aux objectifs suivants :

  • Proactivité : Détecter et signaler les incidents avant que les clients n'appellent
  • Efficacité : Automatiser le traitement des tickets simples et répétitifs
  • Capitalisation : Centraliser et enrichir continuellement la base de connaissances
  • Assistance : Fournir aux techniciens un accès instantané à l'information
  • Sécurité : Garantir un contrôle humain sur toutes les actions critiques

A contrario, les objectifs ne sont pas :

  • De remplacer les techniciens par l'IA
  • D'exécuter des actions critiques sans validation humaine
  • De se substituer aux outils existants (GLPI, Observium, AD)

1.2. Périmètre et composants principaux

La plateforme se compose de cinq éléments principaux :

Composant Description Type
WIBOT Interface conversationnelle principale pour les techniciens, avec 4 modes de chat (Flash, Pro, Code, Agent) Interface
SENTINEL Workflow autonome de surveillance proactive des infrastructures clientes via Observium Workflow
ASSISTANT TICKET Workflow autonome de qualification et résolution automatique des tickets GLPI Workflow
ENRICHISSEUR Workflow batch de capitalisation des connaissances à partir des tickets résolus Workflow
SAFEGUARD Système de sécurité contrôlant toutes les actions critiques avant exécution Sécurité

Ces composants s'appuient sur :

  • RAG (Base de connaissances) : Stockage vectoriel des procédures et documentation
  • MCP (Model Context Protocol) : Connecteurs vers les systèmes externes (GLPI, Observium, AD)
  • LLM Mistral : Modèles d'intelligence artificielle pour le traitement du langage

2. Dossier fonctionnel - WIBOT

2.1. Vue d'ensemble de l'interface

WIBOT est l'interface conversationnelle principale de la plateforme. C'est le point d'entrée pour tous les techniciens de WIDIP. Il permet d'interagir avec l'ensemble du système via une conversation naturelle.

L'interface se compose de :

  • Chat : Zone de conversation avec l'IA (modes Flash, Pro, Code, Agent)
  • Safeguard : Tableau de bord des demandes de validation en attente
  • Knowledge : Gestion de la base de connaissances RAG
  • Supervision : Statistiques d'utilisation et analytics
  • Administration : Gestion des utilisateurs et configuration (réservé aux admins)

2.2. Les 4 modes de chat

WIBOT propose 4 modes de chat adaptés à différents cas d'usage :

2.2.1. Mode Flash

Caractéristique Description
Objectif Réponses rapides aux questions simples
Modèle LLM Devstral Small (rapide, économique)
Cas d'usage "C'est quoi le VLAN pour le site X ?", "Quel est le contact du client Y ?"
Accès RAG Oui, pour enrichir les réponses
Accès MCP Non

2.2.2. Mode Pro

Caractéristique Description
Objectif Requêtes complexes nécessitant une analyse approfondie
Modèle LLM Mistral Large (haute qualité)
Cas d'usage "Explique-moi l'architecture réseau du client Z", "Quelle est la procédure complète pour migrer un serveur ?"
Accès RAG Oui, recherche approfondie dans la base de connaissances
Accès MCP Non

2.2.3. Mode Code

Caractéristique Description
Objectif Conception et optimisation de scripts
Modèle LLM Devstral (spécialisé code)
Cas d'usage "Écris un script PowerShell pour lister les users AD inactifs", "Optimise ce script Python"
Accès RAG Oui, pour les exemples et bonnes pratiques

2.2.4. Mode Agent

Caractéristique Description
Objectif Résolution de problèmes complexes nécessitant des actions
Modèle LLM Mistral Large (raisonnement avancé)
Cas d'usage "Diagnostique pourquoi le client X a des lenteurs", "Trouve tous les tickets ouverts pour le site Y et résume la situation"
Accès RAG Oui, comme outil de recherche
Accès MCP Oui - GLPI, Observium, AD, etc.

Fonctionnement du Mode Agent

L'Agent peut simultanément interroger plusieurs sources :

  • RAG : Procédures, documentation interne
  • GLPI : Tickets, clients, historique
  • Observium : Métriques, alertes, état des équipements

Il croise les informations pour fournir une vision globale et suggère des procédures adaptées au problème détecté.

Évolution prévue : À terme, l'Agent pourra exécuter les procédures suggérées (toujours avec validation Safeguard pour les actions sensibles).

2.3. Accès à la base de connaissances

Dans tous les modes, WIBOT peut accéder à la base de connaissances RAG pour :

  • Enrichir ses réponses avec des informations contextuelles
  • Fournir des procédures documentées
  • Citer des cas similaires déjà résolus

L'accès au RAG est transparent pour l'utilisateur : l'IA recherche automatiquement les informations pertinentes.

3. Dossier fonctionnel - Workflows autonomes

La plateforme dispose de trois workflows autonomes qui fonctionnent en arrière-plan, avec une semi-autonomie contrôlée par le système Safeguard.

3.1. Workflow SENTINEL

3.1.1. Objectif et principe

Sentinel est un workflow autonome qui surveille en continu les infrastructures clientes via Observium. Son objectif est de détecter les pannes AVANT que les clients n'appellent, permettant une gestion proactive du support.

Aspect Description
Fréquence Interrogation toutes les 1 minute
Cache Redis Stocke l'état des équipements pour détecter les pannes persistantes (seuil : 20 min)
Vérification FAI Le technicien humain vérifie manuellement si la panne vient du FAI ou de l'infrastructure gérée par WIDIP
Notifications SMS aux clients (via Safeguard L3), notification technicien
Intégration Peut déclencher l'Assistant Ticket pour créer automatiquement un ticket

3.1.3. Règles métier

  • Seuil de persistance : Une panne n'est considérée comme critique qu'après 20 minutes de DOWN continu
  • Vérification FAI : Le technicien doit vérifier manuellement si la panne provient du FAI ou de l'infrastructure WIDIP
  • SMS clients : L'envoi de SMS nécessite une validation Safeguard L3 (admin requis)
  • Ticket automatique : Le ticket est créé après validation, avec le numéro communiqué au client

Bénéfices

  • Proactivité : Les clients sont informés avant même qu'ils n'appellent
  • Réduction charge support : Moins d'appels entrants car les clients sont déjà au courant
  • Traçabilité : Toutes les pannes sont documentées avec timeline précise
  • Qualification : Distinction claire entre panne FAI et panne infrastructure WIDIP

3.2. Workflow ASSISTANT TICKET

3.2.1. Objectif et principe

L'Assistant Ticket est un workflow autonome qui surveille les nouveaux tickets GLPI toutes les 3 minutes. Il qualifie chaque ticket en niveau de difficulté et peut résoudre automatiquement les tickets simples.

3.2.2. Qualification des tickets

Pour chaque nouveau ticket, l'Assistant Ticket effectue une analyse comprenant :

  • Titre et description du ticket
  • Catégorie
  • Client concerné
  • Historique des tickets similaires (via RAG)

Il attribue ensuite un niveau de difficulté :

Niveau Description Traitement
SIMPLE Ticket dans l'éventail de compétences de l'IA L'IA peut résoudre (avec Safeguard)
MOYEN Ticket nécessitant une analyse plus poussée L'IA peut tenter (avec Safeguard)
COMPLEXE Ticket dépassant les compétences de l'IA Escalade technicien N1
EXPERT Ticket nécessitant une expertise avancée Escalade technicien N2+

3.2.3. Résolution automatique

L'Assistant Ticket peut résoudre automatiquement les types de demandes suivants :

Type de demande Actions de l'IA Niveau Safeguard
Création compte AD Créer l'utilisateur selon template L3 (validation requise)
Reset mot de passe Générer nouveau MDP, envoyer au user L3 (validation requise)
Déblocage compte Unlock dans AD L2 (notification tech)
Ajout groupe AD Ajouter user au groupe demandé L2 (notification tech)
Info procédure Fournir la procédure depuis RAG L0 (auto)
Phase de test : Durant la phase de rodage, la majorité des actions seront en L3 (validation humaine obligatoire) pour garantir la fiabilité du système.

3.2.4. Gestion des tickets non résolus

Les tickets que l'IA ne peut pas résoudre (COMPLEXE ou EXPERT) sont :

  • Flaggés : Stockage de l'ID dans Redis avec tag "IA_NON_RESOLU"
  • Conservés : Pour analyse ultérieure par l'Enrichisseur
  • Escaladés : Restent en attente pour le technicien approprié

Quand un technicien résout un ticket flaggé, l'Enrichisseur analyse la solution pour potentiellement créer une nouvelle procédure.

Bénéfices

  • Réduction temps résolution : Les tickets simples sont traités en minutes
  • Disponibilité 24/7 : L'IA traite même hors heures ouvrées
  • Cohérence : Mêmes procédures appliquées systématiquement
  • Libération techniciens : Se concentrent sur les cas complexes
  • Apprentissage : Les tickets non résolus alimentent l'Enrichisseur

3.3. Workflow ENRICHISSEUR

3.3.1. Objectif et principe

L'Enrichisseur est un workflow batch (quotidien ou hebdomadaire) qui analyse les tickets résolus pour en extraire des procédures. Il transforme l'expérience des techniciens en connaissances réutilisables.

3.3.2. Sources de données

L'Enrichisseur analyse :

  • Tickets clôturés : Du jour ou de la semaine, bien documentés
  • Tickets flaggés "IA_NON_RESOLU" : Maintenant résolus par un technicien

4. Dossier fonctionnel - SAFEGUARD

4.1. Principe et objectifs

Safeguard est le verrou de sécurité qui contrôle TOUTES les actions critiques avant leur exécution. Il garantit qu'aucune action sensible n'est réalisée sans validation humaine appropriée.

Objectifs :

  • Prévenir les erreurs automatisées irréversibles
  • Garantir la traçabilité de toutes les actions
  • Permettre une semi-autonomie contrôlée des workflows
  • Adapter le niveau de contrôle à la criticité de l'action

4.2. Niveaux d'action (L0 à L4)

Niveau Nom Description Comportement
L0 READ_ONLY Lecture seule, aucune modification Exécution automatique
L1 MINOR Modification mineure, réversible Auto si confiance > 80%
L2 MODERATE Impact modéré Notification tech requise
L3 SENSITIVE Action sensible Validation humaine OBLIGATOIRE
L4 FORBIDDEN Action interdite L'IA ne peut JAMAIS exécuter

Exemples par niveau

Niveau Actions
L0 Rechercher un ticket, Voir les métriques Observium, Consulter le RAG
L1 Créer un ticket, Ajouter un suivi
L2 Débloquer un compte AD, Notifier un technicien, Changer statut ticket
L3 Reset mot de passe, Clôturer ticket, Envoyer SMS client, Créer compte AD
L4 Supprimer compte AD, Actions irréversibles sur production

4.3. Niveaux d'accréditation utilisateur (N0 à N4)

Chaque utilisateur possède un niveau d'accréditation qui détermine les actions qu'il peut valider :

Niveau Rôle Peut valider
N0 Lecture seule Aucune validation
N1 Technicien L1
N2 Technicien senior L1, L2
N3 Administrateur L1, L2, L3
N4 Super administrateur L1, L2, L3 + configuration système
Synchronisation : Les niveaux d'accréditation peuvent être synchronisés avec la matrice d'autorisation WIDIP (fichier Excel) pour une gestion centralisée.

4.4. Interface et processus de validation

L'interface Safeguard présente les demandes de validation avec :

  • Workflow source : Quel workflow a généré la demande (Sentinel, Assistant Ticket, Agent...)
  • Niveau requis : L1, L2 ou L3
  • Contexte : Informations détaillées sur la situation
  • Action demandée : Ce que l'IA souhaite faire
  • Justification : Pourquoi l'IA a besoin de faire cette action
