Resume executif

La plateforme WIDIP est un systeme multi-agents IA concu pour le support informatique. Elle combine une interface conversationnelle (WIBOT), trois workflows autonomes (Sentinel, Assistant Ticket, Enrichisseur), un framework de securite (SAFEGUARD) et une base de connaissances evolutive (RAG).

Le projet a debute fin novembre 2025 avec une approche low-code (N8N), puis a ete migre vers une architecture Python en janvier 2026 suite a l'identification des limites de l'approche initiale. Cette evolution technologique constitue une preuve de la demarche experimentale et de l'incertitude technique du projet.

1. Presentation du projet et contexte

1.1 Contexte et enjeux

WIDIP est une entreprise de services informatiques gerant le support technique pour un portefeuille de PME clientes. Le support traditionnel fait face a plusieurs defis structurels :

  • Perte de connaissances : l'expertise des techniciens n'est pas capitalisee et disparait lors des departs
  • Traitement reactif : les clients signalent les pannes, les techniciens ne les detectent pas en amont
  • Taches repetitives : une part significative des tickets concerne des operations simples (reset mot de passe, deblocage compte)
  • Risque d'erreur : les actions sur les systemes critiques (Active Directory, infrastructure reseau) necessitent un controle rigoureux

1.2 Objectifs du projet

Le projet vise a concevoir et developper une plateforme IA repondant a ces defis par :

  1. La capitalisation automatique des connaissances issues de la resolution de tickets
  2. La detection proactive des pannes avant signalement client
  3. La resolution autonome des tickets simples avec validation humaine appropriee
  4. Un framework de securite garantissant le controle humain sur les actions critiques

1.3 Historique et evolution du projet

1.4 Positionnement R&D

Problematique centrale

Le projet ne consiste pas simplement a assembler des composants existants (LLM, base vectorielle, API). Les verrous de recherche resident dans l'orchestration, la fiabilite et les boucles d'apprentissage d'un systeme multi-agents operant sur des systemes de production IT avec des actions pouvant etre irreversibles.

Le pivot N8N ‚Üí Python illustre cette incertitude : l'approche initiale, bien que fonctionnelle pour des cas simples, s'est averee inadaptee pour resoudre les verrous identifies (notamment la gouvernance SAFEGUARD).

Les travaux de recherche portent specifiquement sur :

  • Comment un LLM peut-il extraire des procedures generalisables a partir de texte non structure ?
  • Comment calibrer la confiance d'un LLM pour autoriser des actions IT avec consequences reelles ?
  • Comment concevoir un framework de securite pour un systeme multi-agents distribue ?

3. Operation R&D 1 : Capitalisation automatique des connaissances IT

Composants concernes : Enrichisseur + RAG evolutif Potentiel CIR : Fort

Statut : EN COURS - L'Enrichisseur est en phase de prototypage. Les premiers tests d'extraction de procedures ont ete realises mais le systeme complet n'est pas encore operationnel.

3.1 Problematique scientifique

Comment transformer automatiquement l'historique operationnel non structure d'une PME (tickets, conversations, resolutions) en une base de connaissances fiable, evolutive et exploitable par des agents IA - sans degradation de qualite dans le temps ?

3.2 Verrous techniques identifies

1 Extraction de procedures a partir de texte libre

Les tickets GLPI sont rediges en langage naturel non standardise par des techniciens et des utilisateurs finaux. L'information utile est noyee dans du bruit (formules de politesse, descriptions imprecises, fautes). Le LLM doit identifier un pattern de resolution generalisable a partir de cas individuels, ce qui depasse la simple synthese de texte.

Incertitudes scientifiques
  • Comment distinguer une resolution reellement transposable d'un cas trop specifique ?
  • Comment extraire des etapes de procedure quand le technicien n'a pas documente toutes ses actions ?
  • Quel seuil de recurrence (nombre de tickets similaires) est necessaire pour justifier la creation d'une procedure ?
2 Maintien de la qualite d'une base RAG auto-alimentee

La base de connaissances s'enrichit en continu via l'Enrichisseur. Cela cree un risque de data drift : procedures obsoletes, contradictions entre anciennes et nouvelles procedures, redondances qui polluent les resultats de recherche semantique.

Incertitudes scientifiques
  • Comment detecter qu'une procedure existante est rendue obsolete par une nouvelle ?
  • Comment mesurer la qualite globale du RAG au fil de son enrichissement ?
  • Comment eviter que l'accumulation de documents degrade la precision de retrieval plutot que de l'ameliorer ?
3 Boucle de feedback et cercle vertueux

Le systeme repose sur un cercle : tickets resolus -> procedures generees -> RAG enrichi -> IA plus performante -> meilleure resolution -> plus de matiere pour l'Enrichisseur. La convergence de cette boucle n'est pas garantie.

Incertitudes scientifiques
  • La boucle converge-t-elle reellement (amelioration mesurable) ou diverge-t-elle (accumulation de bruit) ?
  • Comment mesurer l'impact de chaque procedure ajoutee sur la performance globale du systeme ?
  • Quel est le ratio optimal entre procedures generees automatiquement et procedures validees humainement ?

4. Operation R&D 2 : Qualification et resolution autonome de tickets IT

Composants concernes : Assistant Ticket + Sentinel Potentiel CIR : Fort

Statut : EN COURS - L'Assistant Ticket est fonctionnel en mode consultation. La resolution autonome est en cours d'implementation. Sentinel est operationnel pour la detection de pannes.

4.1 Problematique scientifique

Comment un systeme IA peut-il qualifier de maniere fiable la difficulte d'un ticket IT et resoudre les cas simples de facon autonome - tout en garantissant un niveau de confiance calibre et un fallback sur vers l'humain lorsque l'incertitude est trop elevee ?

4.2 Verrous techniques identifies

1 Qualification automatique en 4 niveaux de difficulte

L'Assistant Ticket doit classer chaque ticket entrant en SIMPLE / MOYEN / COMPLEXE / EXPERT. Cette classification determine directement le traitement : resolution IA autonome vs escalade humaine. Une erreur de classification a des consequences operationnelles directes.

Incertitudes scientifiques
  • Comment un LLM peut-il evaluer la difficulte reelle d'un ticket a partir d'une description souvent incomplete ou ambigue ?
  • Comment gerer le cas critique du faux SIMPLE (ticket classe simple mais qui necessitait un expert) ?
  • Comment integrer le contexte historique du client (tickets passes, infrastructure) dans la qualification ?
2 Calibration de l'indice de confiance

Le systeme SAFEGUARD utilise un seuil de confiance (> 80% pour execution automatique en L1). Ce seuil conditionne le degre d'autonomie du systeme. Or, les LLM sont notoirement mal calibres en termes de confiance - ils expriment souvent une haute confiance meme lorsqu'ils se trompent.

Incertitudes scientifiques
  • Comment obtenir un score de confiance fiable d'un LLM pour des actions IT concretes ?
  • Comment calibrer ce score (faire en sorte que "80% de confiance" corresponde reellement a 80% de reussite) ?
  • Le seuil optimal est-il universel ou doit-il varier par type d'action, par client, par complexite ?
3 Resolution autonome avec actions irreversibles

Resoudre un ticket simple implique des actions reelles : reset de mot de passe AD, deblocage de compte, creation d'utilisateur. Ces actions ont des effets concrets et certaines sont irreversibles.

Incertitudes scientifiques
  • Comment valider qu'une procedure RAG retrouvee est applicable au cas precis du ticket courant ?
  • Comment detecter une mauvaise application de procedure avant l'execution de l'action ?
  • Quel protocole de rollback pour les actions partiellement executees en cas d'erreur detectee en cours ?
4 Decision autonome dans le monitoring proactif (Sentinel)

Sentinel doit prendre des decisions en temps reel : identifier les clients impactes par une panne, determiner s'il s'agit d'une panne FAI ou infrastructure, decider quand et qui alerter.

Incertitudes scientifiques
  • Comment l'IA peut-elle distinguer automatiquement une panne FAI d'une panne infrastructure WIDIP sans intervention humaine ?
  • Comment le parsing de hostnames (format CLIENT-SITE-VILLE-DEVICE) peut-il gerer les cas non conformes au pattern standard ?
  • Comment correler automatiquement un equipement Observium DOWN avec les bonnes entites GLPI quand les nomenclatures ne sont pas parfaitement alignees ?

Resultat cle : Echec de l'auto-evaluation de confiance

L'hypothese initiale etait de demander au LLM de s'auto-evaluer. Les tests ont montre que les scores etaient systematiquement superieurs a 85%, meme pour des cas ou le LLM se trompait. Ce phenomene de "surconfiance" est documente dans la litterature et constitue un verrou de recherche reel. Une approche alternative basee sur la similarite RAG est en cours de test.

5. Operation R&D 3 : Gouvernance de securite multi-agents

Composant concerne : SAFEGUARD (framework complet) Potentiel CIR : Fort

Statut : QUASI FINALISE - Le framework SAFEGUARD est operationnel et en production. Les 6 niveaux de securite (L0 a L4_SMS) sont implementes. Bug double-blocage resolu.

5.1 Problematique scientifique

Comment concevoir un framework de gouvernance garantissant la securite d'un systeme multi-agents IA (3 workflows autonomes + 1 agent conversationnel) operant sur des systemes de production IT (Active Directory, ticketing, monitoring reseau) avec des actions pouvant etre irreversibles ?

5.2 Verrous techniques identifies

1 Modele de securite multi-niveaux pour agents IA

Le framework SAFEGUARD definit 6 niveaux de sensibilite (L0 a L4_SMS) croises avec 5 niveaux d'accreditation utilisateur (N0 a N4). Cette matrice determine dynamiquement ce qu'un agent IA peut faire, ce qu'il doit demander, et ce qu'il ne peut jamais faire.

Incertitudes scientifiques
  • Comment definir la bonne granularite de niveaux ? Trop peu = risque de securite, trop = friction operationnelle
  • Comment gerer les cas limites entre deux niveaux ?
  • Le modele est-il stable lorsque de nouveaux outils MCP sont ajoutes, ou necessite-t-il une reevaluation globale ?
2 Coherence dans une architecture distribuee

Le double systeme SAFEGUARD (backend PostgreSQL + MCP Server Python) a genere un probleme de recherche concret : apres approbation humaine cote backend, le MCP Server re-bloquait l'action. La solution (_bypass_safeguard) implique de maintenir la coherence d'etat entre deux systemes independants dans un contexte asynchrone.

Incertitudes scientifiques
  • Comment garantir la synchronisation des niveaux de securite entre deux systemes independants ?
  • Comment prevenir les race conditions entre l'approbation et l'execution dans un flux SSE asynchrone ?
  • Comment auditer de maniere fiable les actions executees via bypass dans un systeme distribue ?
3 Orchestration securisee de workflows autonomes chaines

Sentinel peut declencher l'Assistant Ticket, qui peut generer des actions necessitant des validations SAFEGUARD. Cette chaine cree des scenarios complexes : un workflow autonome declenche un autre workflow qui declenche une action L3 necessitant une validation humaine, pendant que le premier workflow continue de tourner.

Incertitudes scientifiques
  • Comment gerer les dependances entre actions dans une chaine de workflows ?
  • Comment eviter les deadlocks quand un workflow attend une approbation pendant qu'un autre genere de nouvelles demandes ?
  • Comment prioriser les demandes SAFEGUARD quand plusieurs workflows sollicitent simultanement l'humain ?
4 Confiance dynamique et niveau L4_SMS specialise

Le niveau L4_SMS a ete cree specifiquement pour l'alerte SMS client - un cas ou ni le blocage total (L4) ni la simple validation (L3) ne suffisaient. Il integre une interface de selection de contact avec recommandation IA et apprentissage par feedback humain (cache Redis mis a jour si l'humain choisit un contact different).

Incertitudes scientifiques
  • Comment le systeme apprend-il quel contact recommander au fil du temps ?
  • Le score d'activite DirEnt est-il un proxy fiable pour la pertinence du contact ?
  • Comment gerer la degradation de la recommandation quand les contacts changent de poste ?

5.3 Matrice des niveaux SAFEGUARD

Niveau Nom Description Comportement

L1 MINOR Modification mineure, reversible Auto si confiance > 80%
L2 MODERATE Impact modere Notification technicien requise

Preuve d'incertitude : Bug double-blocage

Probleme rencontre : Apres approbation humaine via l'interface frontend, l'action etait a nouveau bloquee lors de l'execution par le MCP Server. Cause : le MCP Server appliquait sa propre verification SAFEGUARD sans savoir que l'action avait deja ete approuvee cote backend.

Solution experimentale : Implementation d'un mecanisme _bypass_safeguard avec token d'approbation transmis du backend vers le MCP Server. Cette solution a ete trouvee apres echec de l'hypothese initiale (polling de base de donnees).

6.3 Analyse des criteres

Nouveaute

Les trois operations portent sur des problemes pour lesquels l'etat de l'art ne propose pas de solution etablie :

  • La capitalisation automatique de connaissances IT par LLM avec boucle de feedback est un domaine emergent
  • La calibration de confiance d'un LLM pour des actions IT operationnelles est un probleme de recherche reconnu
  • Les frameworks de gouvernance multi-agents IA en production sont tres peu documentes

Creativite

Le projet propose des architectures originales :

  • Chaine complete ticket -> procedure -> RAG -> amelioration continue
  • Couplage qualification/resolution avec flagging pour enrichissement ulterieur
  • Modele de securite L0-L4_SMS avec approbation inline SSE et apprentissage du referent

Incertitude

11 verrous techniques ont ete identifies, chacun portant des incertitudes scientifiques non resolues au demarrage du projet. Plusieurs hypotheses ont echoue, demontrant l'incertitude reelle :

  • OR2 : L'auto-evaluation de confiance par le LLM a echoue (scores > 85% non discriminants)
  • OR3 : La synchronisation par polling a echoue (latence, race conditions), conduisant au mecanisme de bypass
  • Pivot N8N ‚Üí Python : L'approche low-code initiale s'est averee inadaptee pour les cas complexes

Systematisation

Point d'attention

C'est le critere le plus faible. Pour renforcer l'eligibilite, il est necessaire de produire :

  • Un etat de l'art formel pour chaque operation (publications, solutions existantes, leurs limites)
  • Des hypotheses formalisees avant chaque experimentation
  • Des protocoles de test avec metriques mesurables
  • Des resultats chiffres (positifs ET negatifs)
  • Un journal de bord R&D avec chronologie des iterations

Transferabilite

Les methodologies developpees sont transposables :

  • La capitalisation automatique s'applique a tout contexte de support (client, maintenance industrielle)
  • La qualification par confiance calibree est utilisable dans tout domaine d'automatisation IA
  • SAFEGUARD est generalisable a tout systeme multi-agents avec actions critiques

7.3 Methodologie de versioning et tracabilite

Approche de gestion des versions

Le projet utilise une methodologie de versioning manuelle par copies incrementales. Chaque version stable est conservee dans un repertoire dedie, permettant de reconstituer l'historique complet des travaux de recherche.

Historique des evolutions majeures

Phase Approche technique Raison du changement Valeur R&D
Phase 1 Architecture N8N (low-code) Prototype initial, validation du concept Exploration des possibilites
Phase 2 Migration vers Python Limites de N8N pour les cas complexes (orchestration multi-agents, SAFEGUARD) Preuve d'incertitude technique - l'approche initiale ne permettait pas de resoudre les verrous identifies
Phase 3+ Iterations Python (v1, v2, v3...) Resolution progressive des verrous techniques Demarche experimentale iterative

Processus de versioning

La methodologie appliquee suit un cycle rigoureux :

  1. Copie de la version stable courante avant toute modification majeure
  2. Developpement et tests sur la copie de travail
  3. Validation fonctionnelle : verification de l'absence de bugs et de regressions
  4. Renommage en version N+1 une fois la stabilite confirmee
  5. Conservation de toutes les versions (y compris les versions intermediaires avec bugs = preuve d'incertitude)

Preuves de tracabilite a constituer

  • Liste des versions : repertoire complet avec dates de creation (attributs fichiers)
  • Journal de bord R&D : document retra√ßant pour chaque version les modifications apportees, les problemes rencontres et les solutions testees
  • Documentation du pivot N8N ‚Üí Python : justification technique du changement d'approche (limites rencontrees, verrous non resolubles)
  • Versions avec bugs conservees : preuves materielles des echecs et iterations (critere d'incertitude)

Valeur probante de cette methodologie

Bien que differente d'un systeme de gestion de versions distribue (Git), cette approche presente une valeur probante pour le CIR :

  • Tracabilite temporelle : les dates de creation/modification des fichiers constituent une preuve horodatee
  • Conservation des echecs : les versions intermediaires avec bugs documentent l'incertitude scientifique
