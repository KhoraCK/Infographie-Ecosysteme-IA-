OPERATION DE RECHERCHE 3 QUASI FINALISE

Gouvernance de securite pour systeme multi-agents IA operationnel

Projet : WIDIP / WIBOT Composant : SAFEGUARD (framework complet) Periode : Novembre 2025 - Fevrier 2026 Potentiel CIR : FORT

1. Problematique scientifique

Comment concevoir un framework de gouvernance garantissant la securite d'un systeme multi-agents IA (3 workflows autonomes + 1 agent conversationnel) operant sur des systemes de production IT (Active Directory, ticketing, monitoring reseau) avec des actions pouvant etre irreversibles ?

Statut actuel : Quasi finalise

Le framework SAFEGUARD est operationnel et en production. Les 6 niveaux de securite (L0 a L4_SMS) sont implementes. Le mecanisme de bypass post-approbation fonctionne. Le systeme d'alerte SMS avec apprentissage du referent est deploye. Des ajustements mineurs sont encore en cours.

2. Etat de l'art

Contexte scientifique

La securite des agents IA autonomes est un domaine emergent. Si des frameworks de guardrails existent pour le filtrage de contenu (toxicite, biais), tres peu de solutions adressent la gouvernance d'agents multi-agents operant sur des systemes IT de production avec actions consequentes. La litterature academique est rare sur ce sujet specifique.

2.1 Solutions existantes et leurs limites

Solution / Approche Description Limites identifiees
LangChain Guardrails Filtrage de contenu et validation de sortie LLM Limite au filtrage textuel, pas de gestion multi-niveaux d'actions IT
Microsoft AutoGen Safety Controles pour agents conversationnels autonomes Oriente conversation multi-agents, peu adapte aux actions IT operationnelles
RBAC classique (Role-Based Access Control) Controle d'acces base sur les roles utilisateur Statique, ne prend pas en compte la confiance de l'IA ni le contexte
Human-in-the-loop simple Validation humaine de toutes les actions Friction excessive, non viable pour operations frequentes, fatigue de validation
Audit trail classique Log des actions pour tracabilite Reactif (post-action), ne previent pas les erreurs

2.2 Positionnement du projet WIDIP

Le framework SAFEGUARD se distingue par plusieurs innovations :

  • Granularite multi-niveaux : 6 niveaux de sensibilite (L0 a L4_SMS) adaptes aux actions IT specifiques
  • Confiance dynamique : le niveau d'autonomie varie selon la confiance de l'IA et le contexte
  • Integration asynchrone SSE : approbation inline sans bloquer les workflows, flux temps reel
  • Apprentissage du referent : le systeme apprend les preferences humaines pour les alertes (L4_SMS)
  • Architecture distribuee : coherence entre backend PostgreSQL et MCP Server Python

Alerte SMS client

Interface specialisee

Niveau Nom Comportement Exemples d'outils MCP
L0 READ_ONLY Execution automatique, pas de validation glpi_get_ticket, observium_get_devices, ad_get_user
L1 MINOR Auto si confiance > 80%, sinon notification glpi_add_comment, glpi_update_status
L2 MODERATE Notification technicien, execution apres delai rag_add_procedure, enrichisseur_submit
L3 SENSITIVE Validation humaine OBLIGATOIRE avant execution ad_reset_password, ad_unlock_account, ad_add_group
L4 DEFERRED Execution differee avec delai d'annulation ad_delete_user, ad_delete_ou, suppression definitive
L4_SMS SMS_ALERT Interface specialisee avec selection du contact referent sms_alert_device_down

Innovation : Niveau L4_SMS

Le niveau L4_SMS a ete cree pour un cas ou ni L3 (validation simple) ni L4 (execution differee) ne convenaient. Une alerte SMS client necessite :

  • Une validation humaine (comme L3)
  • Un choix du contact destinataire (interface specialisee)
  • Une recommandation IA basee sur l'historique (scoring DirEnt)
  • Un apprentissage : si l'humain choisit un autre contact, le cache est mis a jour (TTL 14j au lieu de 7j)

4. Verrous techniques identifies

1 Modele de securite multi-niveaux pour agents IA

Le framework SAFEGUARD definit 6 niveaux de sensibilite (L0 a L4_SMS) croises avec les niveaux d'accreditation utilisateur. Cette matrice determine dynamiquement ce qu'un agent IA peut faire, ce qu'il doit demander, et ce qu'il ne peut jamais faire.

Incertitudes scientifiques (resolues)
  • Comment definir la bonne granularite de niveaux ? ‚Üí 6 niveaux se sont averes suffisants
  • Comment gerer les cas limites entre deux niveaux ? ‚Üí Escalade vers le niveau superieur par defaut
  • Le modele est-il stable avec de nouveaux outils ? ‚Üí Oui, chaque outil MCP declare son niveau

Hypotheses testees

# Hypothese Resultat Date
1 Modele initial a 4 niveaux (L0-L3) Insuffisant - Besoin d'un niveau "differe" (L4) Dec. 2025
2 Ajout niveau L4 (DEFERRED) OK - Actions sensibles differees avec possibilite d'annulation Dec. 2025
3 Creation niveau L4_SMS specialise pour alertes OK - Resout le cas des alertes SMS Jan. 2026
2 Coherence dans une architecture distribuee

Le double systeme SAFEGUARD (backend PostgreSQL + MCP Server Python) a genere un probleme concret : apres approbation humaine cote backend, le MCP Server re-bloquait l'action car il ne savait pas qu'elle avait ete approuvee.

Bug rencontre : Double blocage

Symptome : Apres approbation humaine via l'interface frontend, l'action etait a nouveau bloquee lors de l'execution par le MCP Server.

Cause identifiee : Le MCP Server appliquait sa propre verification SAFEGUARD sans savoir que l'action avait deja ete approuvee cote backend.

Analyse : Probleme de coherence d'etat entre deux systemes independants dans un flux asynchrone (SSE).

Solution implementee : Mecanisme _bypass_safeguard avec token d'approbation transmis du backend vers le MCP Server.

Incertitudes scientifiques (resolues)
  • Comment garantir la synchronisation ? ‚Üí Token de bypass avec validation
  • Comment prevenir les race conditions ? ‚Üí Verrouillage optimiste + idempotence
  • Comment auditer les actions bypass ? ‚Üí Log explicite avec approval_id

Hypotheses testees

# Hypothese Resultat Date
1 Synchronisation via base de donnees partagee (polling) Echec - Latence trop elevee, race conditions Jan. 2026
2 Mecanisme de bypass avec parametre _bypass_safeguard RESOLU - Fonctionne, deploye en production Jan. 2026
3 Orchestration securisee de workflows autonomes chaines

Sentinel peut declencher l'Assistant Ticket, qui peut generer des actions necessitant des validations SAFEGUARD. Cette chaine cree des scenarios complexes ou un workflow declenche un autre workflow qui declenche une action L3 necessitant validation humaine.

Incertitudes scientifiques
  • Comment gerer les dependances entre actions ? ‚Üí File d'attente avec priorites
  • Comment eviter les deadlocks ? ‚Üí Timeout + mode non-bloquant pour workflows
  • Comment prioriser les demandes ? ‚Üí Score de criticite (niveau + anciennete)

Hypotheses testees

# Hypothese Resultat Date
1 Workflows bloquants (attente synchrone de l'approbation) Echec - Bloque les autres surveillances Dec. 2025
2 Mode non-bloquant : workflow continue, action en attente OK - Sentinel continue de surveiller pendant l'attente Jan. 2026
4 Confiance dynamique et niveau L4_SMS specialise

Le niveau L4_SMS a ete cree specifiquement pour l'alerte SMS client. Il integre une interface de selection de contact avec recommandation IA et apprentissage par feedback humain (cache Redis mis a jour si l'humain choisit un contact different).

Incertitudes scientifiques
  • Comment le systeme apprend-il quel contact recommander ? ‚Üí Score d'activite GLPI + historique des choix humains
  • Le score d'activite est-il un proxy fiable ? ‚Üí Oui pour 80% des cas, confirmation humaine sinon
  • Comment gerer les changements de poste ? ‚Üí TTL du cache (7j auto, 14j si confirme)

Hypotheses testees

# Hypothese Resultat Date
1 Score base sur l'activite recente dans GLPI (tickets, connexions) OK - Bonne correlation avec le referent reel Jan. 2026
2 Cache Redis avec TTL variable (7j auto / 14j si confirme humain) OK - Equilibre entre fraicheur et stabilite Jan. 2026
3 Mise a jour du cache si choix humain different de la recommandation OK - Apprentissage effectif observe Fev. 2026

6. Experimentations et resultats

6.1 Protocole de test

SAFEGUARD a ete teste en conditions reelles sur l'environnement de production WIDIP :

  • Periode de test : 6 semaines (mi-decembre 2025 - fin janvier 2026)
  • Volume : ~200 demandes SAFEGUARD traitees
  • Scenarios : actions AD, alertes SMS, ajout procedures RAG
  • Validation : revue manuelle de tous les cas L3+ par technicien senior

6.2 Metriques collectees

Metrique Description Valeur mesuree Objectif
Demandes L0-L1 Actions executees automatiquement ~65% du total > 60%
Demandes L2-L3 Actions necessitant validation ~30% du total < 35%
Demandes L4_SMS Alertes SMS avec selection contact ~5% du total -
Taux d'approbation L3 % de demandes L3 approuvees ~92% > 85%
Temps de reponse humain Delai median entre demande et decision ~3 min < 5 min
Precision recommandation L4_SMS % de fois ou le contact recommande est choisi ~78% > 70%
Near-misses bloques Actions potentiellement dangereuses arretees 3 cas documentes Documenter

6.3 Resultats

Resultats positifs

  • Bug double-blocage resolu : le mecanisme bypass fonctionne parfaitement depuis mi-janvier
  • Equilibre autonomie/securite atteint : 65% d'actions automatiques, 35% avec validation
  • Apprentissage L4_SMS effectif : la precision de recommandation est passee de 60% a 78%
  • 3 near-misses detectes : actions AD inappropriees bloquees par SAFEGUARD L3
  • Aucun incident de securite : zero action non autorisee executee depuis le deploiement

Points d'amelioration identifies

  • Le taux de faux positifs L2 pourrait etre reduit (certaines notifications inutiles)
  • L'interface d'approbation pourrait etre plus ergonomique sur mobile
  • Le temps de reponse humain augmente en dehors des heures de bureau (a surveiller)
