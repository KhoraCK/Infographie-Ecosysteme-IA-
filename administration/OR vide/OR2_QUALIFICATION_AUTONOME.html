OPERATION DE RECHERCHE 2

Qualification et resolution autonome de tickets IT avec indice de confiance calibre

Projet : WIDIP / WIBOT Composants : Assistant Ticket + Sentinel Potentiel CIR : FORT

1. Problematique scientifique

Comment un systeme IA peut-il qualifier de maniere fiable la difficulte d'un ticket IT et resoudre les cas simples de facon autonome — tout en garantissant un niveau de confiance calibre et un fallback sur vers l'humain lorsque l'incertitude est trop elevee ?

2. Etat de l'art

Contexte scientifique

La calibration de confiance des LLM est un probleme de recherche reconnu. Les modeles de langage sont notoirement mal calibres : ils expriment souvent une haute confiance meme lorsqu'ils se trompent. Appliquer ces modeles a des actions IT avec consequences reelles pose des defis specifiques non resolus.

2.1 Solutions existantes et leurs limites

Solution / Approche Description Limites identifiees
Classification supervisee traditionnelle Modeles ML entraines sur tickets etiquetes Necessite annotations massives, peu adaptatif aux nouveaux types
Routage par mots-cles Regles basees sur la presence de termes specifiques Rigide, nombreux faux positifs, maintenance lourde
LLM avec prompt engineering Classification par LLM sans calibration Confiance non fiable, risque d'erreurs sur cas critiques
Temperature sampling Utilisation de la temperature pour estimer l'incertitude Correlation faible avec la precision reelle

2.3 Positionnement du projet

L'approche WIDIP se distingue par :

  • Calibration contextuelle : ajustement du seuil de confiance par type d'action et par client
  • Fallback structure : integration native avec SAFEGUARD pour escalade humaine
  • Feedback loop : les tickets "IA_NON_RESOLU" alimentent l'apprentissage
  • Actions IT reelles : validation dans un contexte operationnel (pas seulement benchmark)

4. Verrous techniques identifies

1 Qualification automatique en 4 niveaux de difficulte

L'Assistant Ticket doit classer chaque ticket entrant en SIMPLE / MOYEN / COMPLEXE / EXPERT. Cette classification determine directement le traitement : resolution IA autonome vs escalade humaine. Une erreur de classification a des consequences operationnelles directes.

Cas critique : le faux SIMPLE

Un ticket classe SIMPLE par l'IA alors qu'il necessitait un expert peut entrainer : action inappropriee sur le systeme client, perte de temps, degradation de la relation client.

Incertitudes scientifiques
  • Comment un LLM peut-il evaluer la difficulte reelle d'un ticket a partir d'une description souvent incomplete ou ambigue ?
  • Comment gerer le cas critique du faux SIMPLE (ticket classe simple mais qui necessitait un expert) ?
  • Comment integrer le contexte historique du client (tickets passes, infrastructure) dans la qualification ?

2 Calibration de l'indice de confiance

Le systeme SAFEGUARD utilise un seuil de confiance (> 80% pour execution automatique en L1). Ce seuil conditionne le degre d'autonomie du systeme. Or, les LLM sont notoirement mal calibres en termes de confiance — ils expriment souvent une haute confiance meme lorsqu'ils se trompent.

Incertitudes scientifiques
  • Comment obtenir un score de confiance fiable d'un LLM pour des actions IT concretes ?
  • Comment calibrer ce score (faire en sorte que "80% de confiance" corresponde reellement a 80% de reussite) ?
  • Le seuil optimal est-il universel ou doit-il varier par type d'action, par client, par complexite ?

3 Resolution autonome avec actions irreversibles

Resoudre un ticket simple implique des actions reelles : reset de mot de passe AD, deblocage de compte, creation d'utilisateur. Ces actions ont des effets concrets et certaines sont irreversibles.

Incertitudes scientifiques

  • Comment detecter une mauvaise application de procedure avant l'execution de l'action ?
  • Quel protocole de rollback pour les actions partiellement executees en cas d'erreur detectee en cours ?

4 Decision autonome dans le monitoring proactif (Sentinel)

Sentinel doit prendre des decisions en temps reel : identifier les clients impactes par une panne, determiner s'il s'agit d'une panne FAI ou infrastructure, decider quand et qui alerter.

Incertitudes scientifiques
  • Comment l'IA peut-elle distinguer automatiquement une panne FAI d'une panne infrastructure WIDIP sans intervention humaine ?
  • Comment le parsing de hostnames (format CLIENT-SITE-VILLE-DEVICE) peut-il gerer les cas non conformes au pattern standard ?
  • Comment correler automatiquement un equipement Observium DOWN avec les bonnes entites GLPI quand les nomenclatures ne sont pas parfaitement alignees ?
